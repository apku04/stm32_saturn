# Session Log: Unique Source Address from STM32 UID

**Date:** 2026-04-23T19:43Z  
**Session:** Scribe consolidation + Yarrick manifest processing

## Summary

Unique source address (node_addr) now derived from factory-programmed STM32 UID, eliminating beacon collision issues between boards.

## Key Findings

**UID Base Address:**
- STM32U073 UID location: **0x1FFF6E50** (NOT 0x1FFF7590, which is STM32U5)
- UID is 96-bit (three 32-bit words)
- Critical difference from STM32U5 family — incorrect address prevents software from reading correct UID

**Implementation:**
- XOR three UID words → uint32_t, then fold bytes → uint8_t
- Range 1–253 (reserve 0, 254, 255)
- Stored in maclayer.c fallback when no DEVICE_ID in flash
- Boot message prints: `[BOOT] node_addr=N`

## Verification

- **Board 2:** Flashed, UID-derived address confirmed
- **Node Address:** 33 (Board 2 ✅)
- **Status:** Both boards now have unique addresses

## Critical Notes

1. **0x1FFF6E50 is STM32U0-specific** — Do NOT use 0x1FFF7590 (causes read of wrong memory)
2. Unique address stable across resets (factory programmed)
3. No user configuration required — works out-of-box
